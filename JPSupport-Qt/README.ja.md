# JPSupport-Qt

Lazarus (FreePascal) の Qt5 / Qt6 版で、ソースエディタ(SynEdit)に日本語(および他のCJK言語)の入力機能を追加するパッチ集です。

[JPSupport](https://github.com/53jouhikone-source/JPSupport)(GTK2版)の姉妹プロジェクトです。

## なぜQt5・Qt6なのか

Lazarusは長年、画面描画の仕組み(「ウィジェットセット」と呼ばれます)としてGTK2を主力としてきました。しかしGTK2は既に開発が終了しており、後継のGTK3は、少なくとも日本語入力の対応という点では、Qt5・Qt6に比べて明らかに遅れています。GTK2にしがみつき続けることは、Lazarus自体の将来にとって望ましくありません。

私たちは、GTK2版JPSupportの開発を経て、この危機感からQt5・Qt6への対応に着手しました。本プロジェクトは、その第一歩です。まだ荒削りな部分は残っていますが(後述)、Qtの入力メソッドAPIは公式にサポートが続く仕組みであり、GTK2版が使っていた既に非推奨の仕組みよりも、はるかに長い寿命が見込めます。「今すぐ完璧」ではなく「これからも育てていける土台」として、Qt5・Qt6版のLazarusに、GTK2版と同等以上の日本語入力体験を持ち込むことを目指しています。

## 実現した機能

Fcitx5 + Mozc環境での動作を前提に、以下を実現しています。

- **確定処理の正確性**: 複数文字の変換結果が正しく反映されます(以前は一部の文字が欠落するバグがありました)
- **入力切り替えキー**: `Ctrl+Space`、`半角/全角`キーの両方が正常に機能します
- **変換候補ウィンドウの追随**: 変換候補の一覧が、今入力している位置のすぐそばに表示されます(以前は画面の決まった位置にしか出ませんでした)
- **変換中の文字表示**: 変換中の文字列が、画面にきちんと表示されます(以前は何も表示されませんでした)
- **文節の色分け表示**: 今編集している文節(区切られた単語のかたまり)が、水色の文字と太い下線でくっきりと表示されます
- **文節移動への追随**: `←`/`→`、`Shift+←`/`→`で文節を移動すると、この色分け表示も正しく追いかけてきます

一般的なLinuxアプリ(Geditなど)と比較しても遜色ない表示品質を確認しています。

## 導入方法

2通りの方法を用意しています。

### 方法1: Dockerで試す(まず見てみたい方向け)

ビルド済みの環境をそのまま起動できます。今お使いのLazarus環境には一切影響しません。パソコンに詳しくない方は、まずこちらで雰囲気を確認することをお勧めします。

```bash
cd docker
./run-jpsupport-qt5-ubuntu.sh   # Qt5版を試す
# または
./run-jpsupport-qt6-ubuntu.sh   # Qt6版を試す
```

初回は環境の準備(Lazarus本体の構築を含む)に時間がかかります(条件によりますが、数分〜数十分程度)。2回目以降は素早く起動します。

### 方法2: 自分のLazarus環境に組み込む(実際に使いたい方向け)

**率直にお伝えします。この方法は、GTK2版JPSupportの「パッケージをインストールするだけ」という手軽さとは異なり、Lazarus本体を自分でビルドし直す必要があります。** パソコンの操作にある程度慣れている方、時間に余裕のある方向けの手順です。手っ取り早く試したいだけなら、まず方法1をお勧めします。

それでも試してみたい方のために、実際に一般ユーザーの目線で最後まで検証した上での、正直な手順と注意点をまとめます。

#### 事前に知っておいていただきたいこと

- **今お使いのLazarusとは別に、新しいLazarus一式をソースからビルドします。** 既存の環境を上書きするわけではなく、並行して共存させる形になります
- **ビルドには専用の開発ツール一式のインストールが必要**で、Debian/Ubuntu系なら`apt`でまとめて入りますが、それなりの容量とビルド時間(Raspberry Piなら数十分程度)がかかります
- **Qtの画面表示を支える部品(ライブラリ)を、システム側で上書きする必要があります。** 通常は既存の環境に影響しませんが、パッケージ管理の仕組み上、行儀の良い操作とは言えません。心配な方は事前にバックアップを取ることをお勧めします
- **設定ファイルの衝突に注意が必要です。** 初回起動時、Lazarusから「既存の設定と衝突する可能性がある」という警告が出ることがあります。**このとき、絶対に「そのまま使う/更新する」を選ばないでください。** 既存のLazarus環境の設定が壊れる可能性があります。詳しくは下記の手順をご覧ください

#### 手順

1. 必要なツール一式をインストールします(Qtの開発環境、日本語入力関連のパッケージなど)。具体的なパッケージ名は`docker/Dockerfile.ubuntu`(Qt5版)、`docker/Dockerfile.qt6.ubuntu`(Qt6版)の該当箇所を参照してください
2. 新しい場所にLazarusのソースコード一式を取得します

```bash
   git clone --branch fixes_4 https://gitlab.com/freepascal.org/lazarus/lazarus.git lazarus-src
```

3. 取得したフォルダに移動し、パッチを適用します(`qt5`、`qt6`、または両方を指定できます)

```bash
   cd lazarus-src
   python3 /path/to/JPSupport-Qt/patches/apply_jpsupport_patches.py qt5
```

4. 画面表示の部品(ライブラリ)を組み込み直します(Qt5の場合の例)

```bash
   cd lcl/interfaces/qt5/cbindings
   qmake Qt5Pas.pro
   make
   sudo cp -P libQt5Pas.so* /usr/lib/aarch64-linux-gnu/   # 環境により配置場所は異なります
   sudo ldconfig
```

5. Lazarus本体をビルドします(この作業は時間がかかります)

```bash
   cd ../../../..
   make bigide LCL_PLATFORM=qt5
```

6. **既存の設定と衝突しないよう、専用の設定場所を指定して起動します。** ここが一番重要なポイントです

```bash
   ./lazarus --pcp=~/.lazarus_jpsupport_qt5
```

   もし`--pcp`を付けずに起動し、「既存の設定と衝突する可能性がある」という警告が出た場合は、必ず「中止(Abort)」を選んでください。そのまま進めると、既存のLazarus環境の設定が上書きされてしまう可能性があります

これ以降は、この`--pcp`オプションを付けた状態で起動すれば、既存のLazarus環境とは完全に独立した、日本語入力対応版として使い続けられます。

## 技術的な補足(開発者向け)

- Lazarus: `fixes_4`ブランチ(バージョン4.8/4.9系、開発時点での最新の安定系統)を対象に開発・検証
- ウィジェットセット(画面描画の仕組み): Qt5、Qt6の両方に対応
- 入力メソッド: Fcitx5 + Mozcで動作確認(他のIMEでの動作は未検証)
- 検証環境: Ubuntu 24.04 (Docker/ARM64、Raspberry Pi 4/5上)、および Debian 12 (Raspberry Pi 4実機、方法2の直接ビルドで検証)

- **なぜ`libQt5Pas`/`libQt6Pas`にC++拡張が必要だったか**: Qtの`QInputMethodEvent::attributes()`(文節区切り・カーソル位置などの情報)が、Lazarus同梱のバインディングライブラリに一切公開されていなかったため、C++側に直接アクセサ関数を追加しました
- **なぜプリエディット文字列をバッファに挿入しないか**: テキストバッファを汚さず、Undo履歴やシンタックスハイライトへの副作用を避けるため、`TPaintBox`によるオーバーレイ描画方式を採用しています
- **`SlotInputMethodQuery`の`Result`について**: `QEvent::InputMethodQuery`は複数の情報を同時に問い合わせるため、一部の問い合わせにしか答えられない場合でも`Result`を無条件に`True`にしてはいけません(Qt自身の他の処理、特に`Qt::ImEnabled`の判定を妨げ、IME自体が起動しなくなります)。詳細はソースコード中のコメントを参照してください

## 既知の制限・未検証事項

- Fcitx5 + Mozc以外の入力メソッド(ibus等)での動作は未検証です
- Qt5/Qt6以外のウィジェットセット(GTK3等)への対応は含まれません(GTK2版は[JPSupport](https://github.com/53jouhikone-source/JPSupport)を参照してください)
- ルビ(Ruby)属性、周辺テキスト(Surrounding Text)関連の機能には対応していません
- 「方法2」の導入手順は、まだ十分に洗練されていません(パッケージ化やインストーラー化は今後の課題です)

## 今後の展望

本プロジェクトの内容は、Lazarus公式へのアップストリーム提案(バグ報告・マージリクエスト)を見据えた検証・叩き台という位置づけでもあります。将来的に本家に取り込まれ、Qt5・Qt6版のLazarusを使う誰もが、追加の作業なしにこの機能を使えるようになることが、最も望ましい解決だと考えています。
